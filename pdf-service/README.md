# pdf-service

Lightweight, production-ready PDF generation microservice built with Node.js, Puppeteer-core, and headless Chromium. Designed to run efficiently inside Docker with minimal resource usage.

---

## Architecture

```
pdf-service/
├── src/
│   ├── server.js       — Express HTTP layer, routing, graceful shutdown
│   ├── browser.js      — Chromium singleton manager with crash recovery
│   └── pdfService.js   — PDF generation logic + concurrency queue
├── package.json
├── Dockerfile          — Multi-stage slim build
├── .dockerignore
└── README.md
```

### Key design decisions

| Concern | Decision |
|---|---|
| **Browser lifecycle** | One Chromium process for the entire service lifetime (singleton). Saves ~150 MB RSS vs. per-request launches. |
| **Page lifecycle** | One new page per request, closed immediately after use. Prevents state leakage between requests. |
| **Concurrency** | `ConcurrencyQueue` caps simultaneous open pages (default: 5). Additional requests queue and wait rather than OOM-crashing the container. |
| **Crash recovery** | `browser.on('disconnected')` triggers automatic relaunch after 2 s back-off. |
| **Resource interception** | Media, font, and WebSocket requests are aborted for URL-based renders to reduce load time and bandwidth. |
| **Base image** | `node:20-slim` (Debian). Alpine breaks Chromium due to musl-libc. |

## Quick Start

### Docker (recommended)

```bash
docker build -t pdf-service .
docker run -p 3000:3000 pdf-service
```

### Local (requires Chromium installed on host)

```bash
npm install
CHROMIUM_PATH=/usr/bin/chromium node src/server.js
```

## API Reference

### `GET /health`

Liveness probe. Does not touch Chromium.

**Response**
```json
{ "status": "ok", "timestamp": "2024-01-01T00:00:00.000Z" }
```

---

### `POST /generate-pdf`

Generate a PDF from HTML or a URL.

**Headers**
```
Content-Type: application/json
```

**Body**

| Field | Type | Required | Description |
|---|---|---|---|
| `html` | string | one of | Raw HTML string to render |
| `url` | string | one of | Public URL to navigate to |
| `filename` | string | no | PDF filename for Content-Disposition (default: `generated.pdf`) |

**Response**

| Header | Value |
|---|---|
| `Content-Type` | `application/pdf` |
| `Content-Disposition` | `inline; filename="generated.pdf"` |
| Body | Raw PDF bytes |

## curl Examples

### From HTML string

```bash
curl -s -X POST http://localhost:3000/generate-pdf \
  -H "Content-Type: application/json" \
  -d '{"html": "<h1 style=\"font-family:sans-serif\">Hello, PDF!</h1><p>Generated by pdf-service.</p>"}' \
  --output hello.pdf
```

### From a URL

```bash
curl -s -X POST http://localhost:3000/generate-pdf \
  -H "Content-Type: application/json" \
  -d '{"url": "https://example.com", "filename": "example.pdf"}' \
  --output example.pdf
```

### Health check

```bash
curl http://localhost:3000/health
# {"status":"ok","timestamp":"..."}
```

### Error response example

```json
{
  "error": "Bad Request",
  "message": "Provide either \"html\" (string) or \"url\" (string) in the request body."
}
```

## Environment Variables

| Variable | Default | Description |
|---|---|---|
| `PORT` | `3000` | HTTP listen port |
| `CHROMIUM_PATH` | auto-detect | Absolute path to Chromium binary |
| `MAX_CONCURRENT_PAGES` | `5` | Max simultaneous Chromium pages |
| `PAGE_LOAD_TIMEOUT_MS` | `30000` | Max ms to wait for page load |
| `PDF_RENDER_TIMEOUT_MS` | `20000` | Max ms to wait for PDF render |
| `BODY_SIZE_LIMIT` | `10mb` | Max request body size |

## High-Concurrency Tuning (100+ req/min)

For higher throughput, tune these two levers:

```bash
docker run -p 3000:3000 \
  -e MAX_CONCURRENT_PAGES=10 \
  -e PAGE_LOAD_TIMEOUT_MS=15000 \
  --memory=1g \
  pdf-service
```

**Resource budget per concurrent page:** ~25–40 MB RSS  
**Recommended `MAX_CONCURRENT_PAGES`:**

| Container Memory | Recommended |
|---|---|
| 256 MB | 3 |
| 512 MB | 5 (default) |
| 1 GB | 10–15 |
| 2 GB | 20–30 |

Requests beyond the cap are queued in-process (zero extra overhead). At sustained 100 req/min with ~2 s average render time, a concurrency of 10 keeps the queue depth near zero.

## Memory Optimization Notes

- **`--single-process`** — renders pages in the browser process rather than spawning a separate renderer process. Saves ~40 MB per browser instance. Acceptable when input is trusted (your own HTML).
- **`--no-zygote`** — disables the zygote helper process (~30 MB saving).
- **`--disable-dev-shm-usage`** — redirects Chromium's shared memory writes to `/tmp`. Essential in Docker where `/dev/shm` is only 64 MB by default, which causes Chromium to crash under load.
- **`pipe: true`** — uses a stdio pipe instead of a WebSocket for the DevTools Protocol, eliminating the extra HTTP server Chromium would otherwise spawn.
- **Request interception** — aborts media/font/WebSocket requests during URL renders to reduce network I/O and page object count.
- **Page-per-request, not tab reuse** — avoids JS heap accumulation across requests at the cost of a small (~5 ms) page creation overhead.
